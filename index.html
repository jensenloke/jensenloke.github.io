Why a Knowledge Management Layer Still Matters

This tutorial uses a generic agent platform support setup to show the practical gap between no KM, search-only retrieval, and a structured KM layer that lets agents operate safely and consistently.

Exchange Operations Scenario

The example is written for an exchange operations context. The Support Agent handles intake and Support CRM workflow; the Coding Agent handles engineering diagnostics, dependency analysis, DB investigation, RCA, and PR handoff.

What the agent is asked

"This report total does not match the dashboard after market close. Please investigate and move this forward."

What matters operationally

  • Should this route to engineering or manual triage?
  • Who owns the next step: Support Agent, Coding Agent, User, or Manager?
  • Which repo, branch, environment, and database lane are valid?
  • What evidence is required before calling the root cause?
  • Who may approve production diagnostics or changes?

What KM Means Here

In this operating model, KM is not just tagged documents. It is the layer that converts data access into disciplined action.

Skills

Task-specific instructions the agent must load before doing specialized work.

Runbooks

Step order, commands, evidence bars, handoff shapes, and stop conditions.

Permissions

Role, user, channel, tool, sandbox, and approval rules.

Workflow State

Ledgers, memory files, active investigations, PR records, and audit trails.

Source Metadata

Tagged source-system data: services, ticket types, repos, entities, owners, and ACLs.

Head-to-Head Output

Same ticket. Same model quality. Different operating knowledge available.

1. No KM

Only the prompt and generic AI knowledge.

Likely AI response
  • "Check recent code changes around report aggregation."
  • "Ask the operations user for reproduction steps."
  • "Query the trade, adjustment, and report snapshot tables."
  • "Create a Jira or GitHub issue if it looks like a bug."
What is missing
  • No knowledge that the Support Agent owns intake and the Coding Agent owns engineering.
  • No reliable direct-route rule for market operations + data discrepancy.
  • No service-lane resolver before repo, branch, or deploy decisions.
  • No production DB approval rule or read-only diagnostic boundary.
  • No required RCA, ledger, or handoff format.
Operational result

Plausible answer, inconsistent execution. The agent can sound helpful while skipping policy, guessing authority, and leaving no durable audit trail.

2. Connectors and Search

SharePoint, Jira, Confluence, Support CRM, and dependency graph data are indexed.

Likely AI response
  • Finds similar report discrepancy tickets and related RCA notes.
  • Finds service map entries for market operations and related repository docs.
  • Finds dependency graph nodes for close report aggregation and trade adjustments.
  • Summarizes possible backend, SQL, and data-pipeline causes.
What is still missing
  • Search does not decide which rule is binding.
  • Search does not enforce who may approve production diagnostics.
  • Search does not require dependency analysis before DB or code work.
  • Search does not update the ticket, collaboration channel, RCA, PR, and memory ledgers in order.
Operational result

Better factual grounding, but the agent still has to infer the operating procedure from retrieved chunks. That inference can vary by prompt, model, and search ranking.

3. Structured KM Layer

Search plus skills, runbooks, permissions, state, and approvals.

Likely AI response
  • The Support Agent classifies market operations + data discrepancy as engineering-routed.
  • The Support Agent moves the ticket to To Dev and creates the engineering issue in the mapped repo.
  • The Coding Agent resolves the service lane before branch, endpoint, DB, or PR decisions.
  • The Coding Agent runs dependency impact analysis before DB queries or code changes.
  • Production diagnostics are read-only, approval-gated, and audit-reasoned.
  • RCA, active investigation ledger, PR evidence, and Support Agent handoff are updated.
Operational result

Consistent execution. The agent knows the procedure, follows authority boundaries, records evidence, and produces a traceable operations and engineering handoff.

What Happens With No KM

No KM does not mean the AI is useless. It means every operational decision is reconstructed from scratch, usually from incomplete prompt context.

1

Role confusion

The agent may investigate directly, assign a User, or ask a Manager, because it does not know the Support Agent to Coding Agent split.

2

Unsafe authority

It may suggest production SQL or status changes without channel, user, or approval context.

3

Weak traceability

It may summarize findings in chat but fail to update the durable RCA, active investigation ledger, ticket comment, or PR evidence.

4

Inconsistent reasoning

One run may search tickets first, another may query the DB first, and another may jump to code changes.

What the KM Layer Makes Repeatable

This is the operating shape the tutorial is trying to make tangible for The stock exchange: KM is the layer that coordinates the work.

01

Intake

The Support Agent reads the operational evidence, captures required fields, creates or updates the Support CRM ticket, and classifies service and request type.

02

Route

Service map plus request-type rules decide engineering route vs manual triage. Market operations + data discrepancy routes to the Coding Agent.

03

Resolve Lane

The Coding Agent resolves service, channel profile, repo, branch base, PR target, environment context, and allowed diagnostic surface.

04

Investigate

Dependency analysis comes first. DB diagnostics are read-only, reasoned, and gated. Findings are recorded as they happen.

05

Handoff

RCA, ticket update, channel report, PR evidence, and Support Agent handoff use a consistent output shape.

What the Stock Exchange's KM Should Contain

If the end state is AI agents that assist daily operations, the KM layer must be more than source tagging.

Practical definition

A KM layer is the operational knowledge substrate that lets agents turn retrieved information into governed work.

It includes source metadata, but also the skills, policies, state, and runbooks that make an agent behave like a trained team member rather than a search box.

Entity Map

Business services, systems, tickets, repos, controls, owners, and environments.

Procedures

Skills and runbooks for intake, triage, diagnostics, PRs, releases, and support replies.

Permissions

Role, user, channel, system, tool, and approval rules. Yes, permissioning belongs in KM.

State

Durable ledgers for active work, handoffs, evidence, audit, and unresolved unknowns.

Observability

What was retrieved, what rule was applied, who approved it, and what artifact changed.
